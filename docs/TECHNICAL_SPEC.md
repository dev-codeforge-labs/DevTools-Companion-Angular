## 1. System Overview

DevTools Companion for Angular is a Firefox WebExtension (Manifest V3) that provides deep runtime introspection of Angular applications by operating across four isolated browser execution contexts. It targets Angular 12 through 18+ running in Firefox 115 or later.

The extension does not require any modifications to the inspected application, does not transmit data to external servers, and operates entirely within the local browser environment.

### 1.1 Execution Context Map

| Context | Script | JavaScript Scope | DOM Access | Angular APIs |
|---|---|---|---|---|
| Page context | `inject/ng-bridge.js` | Shared with the page | Full | Yes (`window.ng.*`) |
| Isolated world | `content_scripts/inspector.js` | Isolated from page | Full (DOM only) | No |
| Service worker | `background/background.js` | Extension background | None | No |
| DevTools panel | `devtools/panel.js` + components | DevTools frame | None | No |

### 1.2 Extension Identity

| Property | Value |
|---|---|
| Extension ID | `angular-inspector@angular-inspector.dev` |
| Manifest version | 3 |
| Minimum Firefox | 115.0 |
| XPI artifact | `dist/angular-inspector.xpi` |
| SHA-256 manifest | `dist/angular-inspector.xpi.sha256` |

## 3. Communication Protocol

All inter-context communication uses the browser's native messaging APIs. No shared memory, no `localStorage`, and no `indexedDB` are used for cross-context data exchange.

### 3.1 Message Flow

```
Angular Page (window.ng, fetch, XHR)
  │
  │  window.postMessage({ __AngularInspector__: true, type, payload })
  ▼
content_scripts/inspector.js  (isolated world)
  │
  │  browser.runtime.sendMessage({ from: 'content', type, payload })
  ▼
background/background.js  (service worker)
  │
  │  port.postMessage({ type, payload, tabId })        [panel → page: port.onMessage]
  ▼
devtools/panel.js  (DevTools panel)
```

### 3.2 Port Naming Convention

When the DevTools panel opens, it connects to the background service worker using a named port:

```
panel-{tabId}
```

Example: `panel-42` for tab ID 42. The background maps each `tabId → port` in a `Map<number, Port>`.

### 3.3 Message Envelope

**Page → Background (via content script):**

```js
{
  __AngularInspector__: true,   // namespace discriminator
  from: 'content',
  type: string,                  // event name (see §3.4)
  payload: object                // event-specific data
}
```

**Background → Panel (via port):**

```js
{
  type: string,
  payload: object,
  tabId: number
}
```

**Panel → Page (via port → background → content script → window.postMessage):**

```js
{
  to: 'page',
  cmd: string,                   // command name (see §3.5)
  payload: object
}
```

**Page command envelope (received by ng-bridge.js):**

```js
{
  __AngularInspectorCmd__: true,
  cmd: string,
  payload: object
}
```

### 3.4 Event Catalogue (Page → Panel)

| Event type | Trigger | Key payload fields |
|---|---|---|
| `ANGULAR_DETECTED` | Angular globals found | `version: 'ivy' \| 'legacy'`, `mode: 'development' \| 'production'` |
| `ANGULAR_NOT_FOUND` | Detection timed out (10 s) | — |
| `COMPONENT_TREE` | Tree refresh completed | `tree: ComponentNode[]` |
| `HTTP_REQUEST` | XHR or fetch completed | `entry: HttpEntry` |
| `HTTP_LOG` | Full log requested | `entries: HttpEntry[]` |
| `STORE_DETECTED` | Store library identified | `type: 'ngrx' \| 'akita' \| 'ngxs'` |
| `STORE_ACTION` | Action dispatched | `action: object`, `state: object` |
| `NAVIGATION` | URL changed | `url: string`, `time: number` |
| `SOURCE_MAPS_FOUND` | Source map discovery complete | `count: number`, `scripts: string[]` |
| `CD_CYCLE` | Change detection cycle recorded | `time`, `duration`, `trigger`, `components` |
| `SUBSCRIPTION_LEAK` | Leaked subscription detected | `component: string`, `count: number` |
| `PROP_UPDATED` | Inline property edit applied | `componentId`, `propPath`, `value` |
| `PROP_UPDATE_ERROR` | Inline property edit failed | `error: string` |

### 3.5 Command Catalogue (Panel → Page)

| Command | Effect |
|---|---|
| `REFRESH_TREE` | Force a full component tree re-scan |
| `HIGHLIGHT` | Show the red overlay on the component's host element |
| `HIDE_HIGHLIGHT` | Remove the overlay |
| `SET_PROPERTY` | Set a component `@Input()` value and trigger `applyChanges()` |
| `GET_HTTP_LOG` | Return the full in-memory HTTP request log |

## 4. Module Specifications

### 4.1 ng-bridge.js

Runs in the page's JavaScript context. This is the only script with direct access to Angular runtime APIs.

#### Angular Detection Algorithm

```
1. Check window.ng && typeof window.ng.getComponent === 'function'  → 'ivy'
2. Check typeof window.getAllAngularRootElements === 'function'      → 'legacy'
3. If neither found: schedule retry in 2 000 ms (max 5 attempts = 10 s)
4. After 5 failures: emit ANGULAR_NOT_FOUND
```

#### Component Tree Walk

```
getRoots()
  → window.getAllAngularRootElements() or [document.querySelector('[ng-version]')]
  → for each root: buildNode(el, null, depth=0)

buildNode(el, parentId, depth)
  → if depth > 30: return null  (cycle / infinite DOM guard)
  → id = el.__aiId || assign random base-36 id
  → comp = window.ng.getComponent(el)
  → compDef = comp.constructor['ɵcmp'] || comp.constructor['ɵdir']
  → extract: name, selector, inputs, outputs, cdStrategy
  → props = getPublicProps(comp)  (keys not starting with __ or ɵ, max 30, depth ≤ 4)
  → children = el.children.map(child → buildNode(child, id, depth+1)).filter(Boolean)
  → return ComponentNode
```

#### HTTP Interception

Two independent patches applied at bridge injection time (before Angular bootstraps):

**XHR patch:** replaces `window.XMLHttpRequest` with a constructor wrapper that intercepts `open()`, `send()`, and `setRequestHeader()` calls and attaches a `loadend` listener.

**Fetch patch:** replaces `window.fetch` with an `async` wrapper that captures request metadata, awaits the response, clones it, and reads the body asynchronously to avoid consuming the original response stream.

Both patches store entries in a module-level `httpLog` array and emit an `HTTP_REQUEST` event per entry. The `responseBody` is capped at 50 000 characters.

#### Object Serialisation (`safeStringify`)

Recursive serialisation with the following rules to avoid `postMessage` failures:

| Value type | Serialisation |
|---|---|
| `null` / primitive | Passed as-is |
| `function` | `'[Function]'` |
| `HTMLElement` | `'[tagname]'` |
| `Array` | First 20 items recursively (depth ≤ 4) |
| `object` | First 30 own enumerable keys recursively (depth ≤ 4) |
| Throws on access | `'[Error]'` |
| Exceeds depth | `'[...]'` |

#### DOM Observer

A `MutationObserver` on `document.body` (childList, subtree) triggers a debounced tree refresh (150 ms debounce) on every DOM mutation to keep the component tree live.

### 4.2 inspector.js (content script)

Runs in the isolated world. Its sole responsibilities are:

1. **Inject `ng-bridge.js`** into the page context via a `<script src="...">` element using `browser.runtime.getURL()`. The script is appended to `<head>` (or `<html>`) and self-removes after loading.
2. **Bridge page → background:** listen to `window.message` events with `__AngularInspector__ === true` and relay via `browser.runtime.sendMessage()`.
3. **Bridge background → page:** listen to `browser.runtime.onMessage` for messages with `to === 'page'` and relay via `window.postMessage()` with the `__AngularInspectorCmd__` namespace.

### 4.3 background.js (service worker)

Stateless message broker with a single piece of mutable state: `Map<tabId, Port>`.

```
onConnect(port)
  → if port.name starts with 'panel-': register port in panelPorts map
  → port.onMessage: forward { to: 'page' } messages to browser.tabs.sendMessage(tabId)
  → port.onDisconnect: remove from map

onMessage(message, sender)
  → if message.from === 'content': forward to panelPorts.get(sender.tab.id)
```

### 4.4 panel.js

The panel orchestrator manages:

- **Tab routing** — `activateTab(name)` shows/hides tab panels and calls the corresponding module's `render(containerEl)` method.
- **Background port** — connects via `browser.runtime.connect({ name: 'panel-{tabId}' })` and reconnects automatically on disconnect (1 s delay).
- **Settings** — loaded from `browser.storage.local` on init, written back on save.
- **Message dispatch** — `handleMessage(msg)` routes incoming events to the appropriate tab module.
- **Public API** — exposes `window.Panel` for cross-module calls (`sendToPage`, `getSettings`, `applySettings`, `resetSettings`, `enableDebugMode`).

### 4.5 Tab Components

Each tab component exports a plain object on `window` with at minimum a `render(containerEl)` method. The panel calls `render` every time the tab is activated, passing the empty container `<div>`. Components maintain their own internal state across tab switches.

| Module | Export | Key methods |
|---|---|---|
| `tree.js` | `window.TreeTab` | `render`, `setData`, `copyState` |
| `http.js` | `window.HttpTab` | `render`, `addEntry`, `setEntries` |
| `store.js` | `window.StoreTab` | `render`, `setStoreType`, `addAction` |
| `router.js` | `window.RouterTab` | `render`, `setCurrentRoute`, `addNavEvent`, `setRouteConfig`, `clearHistory` |
| `di.js` | `window.DiTab` | `render`, `setServices`, `setRootProviders` |
| `performance.js` | `window.PerformanceTab` | `render`, `addCycle`, `addLeak`, `clearData`, `exportData` |
| `sources.js` | `window.SourcesTab` | `render`, `setSourceFiles`, `setNoSourceMaps`, `jumpToLine` |

## 5. Data Models

### 5.1 ComponentNode

```typescript
interface ComponentNode {
  id:          string;           // Random base-36 ID assigned to the DOM element
  parentId:    string | null;
  name:        string;           // Constructor class name
  selector:    string;           // CSS selector from ɵcmp.selectors
  inputs:      Record<string, unknown>;   // @Input() current values
  outputs:     string[];                  // @Output() EventEmitter names
  props:       Record<string, unknown>;   // Other public properties
  cdStrategy:  'Default' | 'OnPush';
  services:    ServiceInfo[];    // Injected services (best-effort)
  children:    ComponentNode[];
  tagName:     string;           // e.g. 'app-root'
}
```

### 5.2 HttpEntry

```typescript
interface HttpEntry {
  id:              string;       // Random base-36 ID
  method:          string;       // 'GET' | 'POST' | 'PUT' | 'DELETE' | 'PATCH' | ...
  url:             string;
  startTime:       number;       // Date.now() ms
  endTime:         number;
  duration:        number;       // endTime - startTime
  status:          number;       // HTTP status code (0 if errored before response)
  requestHeaders:  Record<string, string>;
  responseHeaders: Record<string, string>;
  requestBody:     unknown;      // Serialised via safeStringify
  responseBody:    string;       // First 50 000 chars of response text
  error?:          string;       // Network error message if fetch threw
}
```

### 5.3 ActionEntry (Store)

```typescript
interface ActionEntry {
  index:   number;   // Position in the actions array
  type:    string;   // Action type string (e.g. '[Hero] Load Heroes')
  payload: unknown;
  time:    number;   // Date.now() ms
}
```

### 5.4 NavigationEvent

```typescript
interface NavigationEvent {

  time:   number;
```

### 5.5 CdCycle

```typescript
interface CdCycle {
  time:       number;    // Date.now() ms when cycle started
```


## 6. Security Model

### 6.1 Content Security Policy

The extension complies with Firefox WebExtension CSP requirements. No `eval()`, no `new Function()`, no inline event handlers in panel HTML. All scripts are loaded from extension-local files.

### 6.2 Script Injection

`ng-bridge.js` is injected using the `src` attribute of a `<script>` element pointing to `browser.runtime.getURL('inject/ng-bridge.js')`. This is the only safe way to run code in the page context without triggering CSP violations on pages with strict `script-src` directives, because the script URL is extension-origin, not inline code.

### 6.3 postMessage Namespace Isolation

All `window.postMessage` calls use discriminator properties to prevent collisions with messages from the page itself or other extensions:

- Page → content: `{ __AngularInspector__: true }`
- Background → page (via content): `{ __AngularInspectorCmd__: true }`

Content script listeners verify `e.source === window` before processing.

### 6.4 Data Isolation

- No user data is sent to external servers. All data lives in browser memory and `storage.local`.
- `storage.local` persists only `ExtensionSettings`. No component state, HTTP bodies, or source code is ever persisted to disk by the extension.
- HTTP response bodies are capped at 50 000 characters in memory and are discarded when the log is cleared or the tab is closed.
- Storing sensitive HTTP bodies can be disabled in Settings (`httpCaptureBody: false`).

### 6.5 Minimum Permission Principle

The extension does not request `tabs`, `cookies`, `history`, `bookmarks`, `downloads`, `notifications`, or any other sensitive permission beyond what is listed in §2.2.

## 7. Build System

### 7.1 build.js

A zero-dependency Node.js 18+ script (`build.js`) handles the full build pipeline.

#### Icon Generation

Generates pixel-perfect 48×48 and 96×96 RGBA PNGs without any native dependencies using:

1. **Pixel rasterisation** — fills a `Uint8Array` with RGBA values by rasterising: a rounded-rect clip mask (18% corner radius), an outer equilateral triangle (circumradius 36% of icon size, colour `#DD0031`), and an inner triangle cutout (54% of outer, colour `#1A1A2E`) to produce the Angular-style logo silhouette.
2. **PNG encoding** — builds a valid PNG binary (signature + IHDR + IDAT + IEND chunks) using Node.js `zlib.deflateSync` for the IDAT payload and a pure-JS CRC-32 lookup table for chunk checksums.

#### XPI Packaging

Collects all extension files recursively (excluding `build.js`, `package.json`, `dist/`, `.git/`, `.claude/`, `node_modules/`, and `icons/generate-icons.html`), then writes a valid ZIP archive using a pure-JS ZIP writer:

- Compression method: **stored (0)** — no compression. Browser extensions load fast even uncompressed, and stored files eliminate the `zlib` dependency for packaging.
- Encoding: UTF-8 file names (flag `0x0800`).
- Outputs: `dist/angular-inspector.xpi` and `dist/angular-inspector.xpi.sha256`.

#### npm Scripts

| Script | Command |
|---|---|
| `npm run build` | `node build.js` — icons + XPI |
| `npm run icons` | `node build.js icons` — icons only |
| `npm run package` | `node build.js package` — XPI only |
| `npm run lint` | `web-ext lint` — Mozilla linter |
| `npm run sign` | `web-ext sign` — AMO submission via API |

### 7.2 web-ext (dev dependency)

`web-ext` (Mozilla's official extension toolchain) is used for linting and AMO signing. It is the only dev dependency. Runtime code has zero npm dependencies — all code is vanilla JS loaded directly by the browser.

## 9. Performance Constraints

| Constraint | Limit | Enforcement |
|---|---|---|
| Component tree depth | 30 levels | Hard limit in `buildNode` recursion guard |
| Object serialisation depth | 4 levels | `safeStringify` depth parameter |
| Array serialisation length | 20 items | `safeStringify` array slice |
| Object key serialisation count | 30 keys | `safeStringify` key loop counter |
| HTTP response body capture | 50 000 chars | `responseText.slice(0, 50000)` |
| HTTP log max entries | 500 (default) | Configurable in Settings |
| CD cycle history | 1 000 entries | Module-level array with `shift()` on overflow |
| CD chart visible bars | 60 | `slice(-60)` before render |
| Tree refresh debounce | 150 ms | `clearTimeout` / `setTimeout` in MutationObserver callback |
| Angular detection retries | 5 (× 2 s = 10 s) | Configurable in Settings |
| Component tree re-scan on navigation | 300 ms delay | Allows Angular to complete routing before scan |

## 10. Known Technical Limitations

### 10.1 Production Mode

Angular 9+ tree-shakes the `ng.*` global API in production builds. The following features are unavailable without explicitly calling `enableDebugTools()`:

- Component tree inspection (relies on `ng.getComponent`, `ng.getDirectives`)
- Inline property editing (relies on `ng.applyChanges`)
- DI inspection (relies on `ng.getInjector`)

Features that remain functional in production: HTTP Monitor, Router (URL-based), Source Maps, Performance (limited).

### 10.2 Zoneless Angular (18+)

Angular 18 introduced optional Zone.js-less change detection. The CD profiler hooks (`ApplicationRef.tick` wrapping) depend on Zone.js patching of async APIs. In zoneless apps, cycle recording may not trigger automatically. HTTP Monitor and component tree updates via MutationObserver remain unaffected.

### 10.3 Cross-Origin Iframes

Browser security restrictions prevent the content script from accessing the JavaScript context of cross-origin iframes. Angular components rendered inside a cross-origin iframe are invisible to the extension.

### 10.4 Minified Bundles Without Source Maps

Property names in minified bundles are mangled (e.g., `t`, `n`, `e`). The component tree displays mangled names, and source navigation is unavailable. Enable source maps in the build configuration to restore readable names.

### 10.5 WebSocket Traffic

The HTTP Monitor intercepts only `XMLHttpRequest` and `fetch`. WebSocket connections and Server-Sent Events are not captured.

### 10.6 Pre-Panel HTTP Requests

The `ng-bridge.js` interceptor is installed when the content script runs (`document_start`). HTTP requests fired before the extension's content script executes (e.g., in Web Workers or Service Workers) are not captured.

### 10.7 Store Detection

NgRx store detection relies on the presence of `window.__REDUX_DEVTOOLS_EXTENSION__`. Applications that use NgRx but have disabled or not installed Redux DevTools may not be auto-detected. Akita and NGXS detection requires the respective globals (`window.akita`, `window.ngxs`) to be exposed by the application.
